# Памятка: Урок 2.20.5 (часть 5/10)
## Самое крутое TODO-приложение (с голосовым вводом)

## 1) Что добавилось в части 5

Главное изменение этого этапа: задачи теперь можно редактировать прямо в списке.

На текущем этапе у вас реализовано:
- inline-редактирование текста задачи;
- inline-редактирование дедлайна;
- сохранение изменений по кнопке;
- автосохранение при клике вне формы редактирования;
- синхронизация изменений с API и `localStorage`.

Это важный шаг: приложение перестаёт быть только списком с добавлением и удалением, и становится полноценным редактором задач.

## 2) Что изменилось в архитектуре

- В [src/App.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/App.jsx) появился обработчик `handleUpdata`.
- В [src/components/TodoItem.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/TodoItem.jsx) появились:
  - `isEditing` для режима редактирования;
  - `editText` и `editDeadline` для локальной формы;
  - `editFormRef` для отслеживания клика вне формы;
  - `handleSave` для сохранения изменений.

Теперь поток данных такой:
1. Пользователь делает двойной клик по задаче.
2. `TodoItem` переключается в режим редактирования.
3. Пользователь меняет текст и/или дедлайн.
4. `TodoItem` вызывает `onUpdata(id, newText, newDeadline)`.
5. `App` обновляет `todos`, отправляет `PUT` на сервер и сохраняет результат в `localStorage`.

## 3) Как работает редактирование

В [src/components/TodoItem.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/TodoItem.jsx):
- двойной клик по задаче включает `isEditing`;
- в режиме редактирования вместо текста показывается форма;
- `Enter` вызывает сохранение;
- клик вне формы тоже вызывает сохранение;
- после сохранения `isEditing` становится `false`.

Технически это сделано через связку:
- `useState` для локальных полей формы;
- `useRef` для ссылки на блок формы;
- `useEffect` для подписки на глобальный `click`;
- `useCallback` для стабильной функции `handleSave`.

Это хороший и типичный паттерн для inline-edit UI.

## 4) Что важно понять из части 5

- Локальное состояние формы в `TodoItem` нужно, потому что редактирование происходит прямо внутри карточки.
- Основное состояние `todos` всё равно остаётся в `App`, и это правильно.
- `TodoItem` отвечает только за UI редактирования, а `App` отвечает за реальные данные и синхронизацию.
- Режим редактирования лучше отделять от обычного режима отображения через явный флаг (`isEditing`).

## 5) Почему это важно перед голосовым вводом

Голосовой ввод не только создаёт задачи, но и потенциально может менять их текст.
Чтобы позже добавить voice edit, уже нужна готовая механика редактирования.

На этом этапе база уже есть:
- есть локальная форма редактирования;
- есть центральный обработчик обновления задачи;
- есть синхронизация обновления с сервером.

Значит в следующих частях голос можно будет встроить не только в создание, но и в изменение задачи.
