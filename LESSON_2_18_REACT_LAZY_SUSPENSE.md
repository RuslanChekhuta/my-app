# Урок 2.18. React.lazy + Suspense — памятка

## 1) Суть темы

`React.lazy` нужен для ленивой загрузки компонентов (код грузится не сразу, а по требованию).

`Suspense` показывает запасной интерфейс (`fallback`) на время загрузки lazy-компонента.

Практический эффект:
- быстрее первый рендер;
- меньше начальный размер JS;
- тяжелые части UI загружаются только когда пользователь до них дошел.

## 2) Базовый шаблон

```jsx
import { lazy, Suspense } from "react";

const HeavyComponent = lazy(() => import("./HeavyComponent"));

function Page() {
  return (
    <Suspense fallback={<div>Загрузка...</div>}>
      <HeavyComponent />
    </Suspense>
  );
}
```

## 3) Как это реализовано у вас сейчас

В `App.jsx` уже сделано правильно:
- `lazy(...)` объявлен на верхнем уровне модуля;
- `TextComponent` и `TodoComponent` показываются условно;
- каждый lazy-компонент обернут в `Suspense`.

Фрагмент из вашего текущего кода:

```jsx
import { lazy, Suspense, useState } from "react";

const TodoComponent = lazy(() => import("../TodoComponent.jsx"));
const TextComponent = lazy(() => import("../TextComponent.jsx"));
```

Это корректная схема для урока 2.18.

## 4) Важное разделение загрузок (на примере Todo)

В вашем `TodoComponent` происходит отдельная загрузка данных через `fetch`.

Значит есть 2 стадии:
- `Suspense fallback` в `App.jsx` — ожидание загрузки **кода компонента**;
- `isLoading` в `TodoComponent.jsx` — ожидание загрузки **данных с API**.

Обе стадии нужны и не заменяют друг друга.

## 5) Типичные ошибки

- объявлять `lazy` внутри функции компонента;
- забывать `Suspense` вокруг lazy-компонента;
- делать слишком тяжелый `fallback`;
- использовать `React.lazy` с именованным экспортом без адаптера импорта.

## 6) Чеклист для себя

- `lazy` вынесен на уровень модуля;
- lazy-компонент всегда внутри `Suspense`;
- `fallback` простой и понятный;
- загрузка кода и загрузка данных разделены логически.
