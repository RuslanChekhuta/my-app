## 1) В чем суть версии v3

Если `v2` делала приложение offline-first, то `v3` делает следующий шаг: приложение начинает учитывать конфликты между локальными и серверными данными.

Главная идея этой версии:
- хранить не только очередь локальных действий;
- помнить, на какой версии задачи действие было основано;
- понимать, что на сервере данные могли уже измениться;
- решать, что делать при расхождении версий;
- показывать пользователю состояние очереди синхронизации в интерфейсе.

Это уже заметно взрослее, чем просто "складывать операции в очередь".

### Ремап временных id после пересоздания

- на сервере задача уже отсутствует;
- приложение создаёт её заново;
- связанные офлайн-действия надо перевести на новый server id.

Это очень практичное улучшение: без такого ремапа очередь легко ломается.

## 3) Как теперь работает conflict-aware sync

В [useTodoManagement.js](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/hooks/useTodoManagement.js) синхронизация стала заметно умнее.

### Сценарий `create`

Если действие типа `create`:
- локальная временная задача отправляется на сервер;
- после ответа server id подставляется в локальный список;
- если в очереди остались следующие действия для этой же задачи, они смогут быть перевязаны на новый id.

### Сценарий `update`

Если действие типа `update`:
- сначала приложение делает `fetchTodoById`;
- если серверная задача не найдена, она либо создаётся заново, либо отбрасывается по выбранной стратегии;
- если задача есть, проверяется `hasTodoConflict(serverTodo, baseTodoSnapshot)`;
- при конфликте приложение сообщает пользователю, какая версия принята;
- затем sync завершается по выбранной стратегии.

То есть стратегия разрешения конфликтов сейчас такая:
- обнаружить конфликт;
- уведомить пользователя;
- применить выбранную стратегию (`local wins` или `server wins`).

### Сценарий `delete`

Если действие типа `delete`:
- сначала проверяется, существует ли задача на сервере;
- если задачи уже нет, удаление считается логически завершённым;
- если задача есть и её серверное состояние отличается от локального снапшота, приложение сообщает о конфликте и применяет выбранную стратегию.

Теперь это уже не только жёстко зашитый `local wins`, а переключаемая стратегия синхронизации.

## 6) Что важно понять из версии v3

- Offline-first без учёта конфликтов подходит только для очень простых случаев.
- Как только данные могут меняться и локально, и на сервере, нужен хотя бы базовый механизм сравнения версий.
- `updatedAt` и снапшоты — простой, но очень полезный способ понимать, был ли конфликт.
- Пользователю важно не только сохранить изменения локально, но и видеть, что они ещё не синхронизированы.

Это уже шаг от "мобильной привычки" к настоящему data-consistency мышлению.

## 7) Почему v3 сильнее v2

`v2` решала проблему отсутствия сети.
`v3` начинает решать проблему расхождения данных.

Разница принципиальная:
- в `v2` приложение умело дождаться сети;
- в `v3` приложение пытается понять, что произошло с серверной версией за это время;
- и уже на этой основе решает, как завершить sync.

Это гораздо более зрелый уровень для учебного проекта.

## 9) Технические замечания по текущему состоянию

- Сейчас доступны `local wins` и `server wins`, но всё ещё нет ручного выбора по конкретному конфликту.
  Это уже лучше базовой версии, но до production-grade conflict resolution ещё далеко.

- Синхронизация всё ещё идёт по одному действию за цикл эффекта.
  Это упрощает логику, но может замедлять обработку длинной очереди.

- `fetchTodoById` перед `update` и `delete` добавляет дополнительные сетевые запросы.
  Это оправдано для обучения и надёжности, но может быть дорогим для больших объёмов.

- Подробная панель очереди уже помогает понять состояние sync, но при длинной очереди может понадобиться пагинация или сворачивание.
