## 3) Как сейчас работает голосовой ввод

1. Пользователь нажимает кнопку микрофона.
2. Если `recognition` инициализирован, вызывается `recognition.start()`.
3. `isListening` становится `true`.
4. Web Speech API начинает отдавать результаты через `onresult`.
5. Промежуточный и финальный текст собираются и подставляются в `text`.
6. Пользователь снова нажимает микрофон для остановки.
7. `recognition.stop()` завершает запись, а итоговая строка остаётся в input.
8. Затем задачу можно отправить обычной кнопкой `+`.

Это хороший UX-подход: голос заполняет форму, а не обходит существующую логику `onAdd`.

## 4) Что важно понять из части 9

- Голосовой ввод встроен в уже существующую форму, а не создаёт отдельный поток данных.
- Это правильно, потому что вся текущая логика добавления задачи уже готова:
  - дедлайн,
  - `onAdd`,
  - API,
  - `localStorage`.
- Web Speech API даёт не только итоговый текст, но и промежуточные результаты (`interimResults`).
- `useRef` здесь используется правильно как буфер для финального текста без лишних ререндеров.

## 5) Почему это хороший шаг для архитектуры

Текущая реализация не ломает предыдущую структуру:
- `AddTodo` отвечает только за ввод;
- `onAdd` по-прежнему отвечает за создание задачи;
- `useTodoManagement` и `useTodoActions` не знают, был текст введён руками или голосом.

Это сильное архитектурное решение: голосовой ввод стал лишь новым источником текста, а не отдельной системой задач.

## 7) Технические замечания по текущему коду

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) `recognition` хранится в `useState`, а объект распознавания создаётся внутри `useEffect` с зависимостью от `isListening`.
  Это может приводить к повторному созданию `SpeechRecognition` и запутанному жизненному циклу. Для такого объекта надёжнее `useRef` и инициализация один раз при маунте.

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) `onend` замыкает значение `isListening`, поэтому поведение автоперезапуска записи может быть нестабильным.

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) внутри `onresult` остались отладочные `console.log`.

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) при `handleSubmit` используется `setShowDeadlineInput("")` вместо `false`.

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) нет явного состояния "браузер не поддерживает Web Speech API".
  Сейчас при отсутствии поддержки кнопка просто не даст результата.

- В [AddTodo.jsx](D:/Ruslan/Projects/Web-garage/tasks/my-app/src/components/AddTodo.jsx) есть закомментированный код для управления курсором. Это признак, что логика компонента начинает разрастаться и уже просится в отдельный hook вроде `useSpeechRecognition`.

## 8) Что запомнить из урока

- Голосовой ввод лучше встраивать в существующую форму, а не строить отдельный UI-поток.
- Web Speech API удобно использовать через `SpeechRecognition` или `webkitSpeechRecognition`.
- Для промежуточных результатов полезны `continuous` и `interimResults`.
- Объекты браузерных API с жизненным циклом лучше хранить в `ref`, а не в обычном state.
- После подключения voice input особенно важно сохранять ясную архитектуру.
