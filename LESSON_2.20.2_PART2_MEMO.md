# Памятка: Урок 2.20.2 (часть 2/10)
## Самое крутое TODO-приложение (с голосовым вводом)

## 1) Что изменилось в части 2

На этом этапе приложение стало более «реальным» по структуре данных задачи.
Теперь каждый todo содержит:
- `id`
- `text`
- `completed`
- `createdAt`
- `deadline`
- `order`

Также добавлены:
- управление дедлайном в форме создания задачи;
- отображение даты создания и дедлайна в карточке задачи;
- подсветка просроченного дедлайна (`красный`), если задача не выполнена.

## 2) Что важно в текущей архитектуре

- `src/App.jsx`
  - хранит `todos` как единый источник истины;
  - добавляет задачу через `onAdd(text, deadline)`;
  - переключает выполнение через `onToggleComplete(id)`;
  - удаляет задачи через `onDelete(id)`.

- `src/components/AddTodo.jsx`
  - контролирует `text` и `deadline`;
  - позволяет показать/скрыть поле `datetime-local`;
  - передаёт данные в `onAdd`.

- `src/components/TodoItem.jsx`
  - больше не хранит `isCompleted` локально;
  - отображает состояние из `todo.completed`;
  - форматирует `createdAt` и `deadline` через `toLocaleString("ru-RU")`.

## 3) Почему это хороший шаг перед голосовым вводом

Голосовой ввод лучше подключать, когда:
- данные уже централизованы в `App`;
- форма имеет явные контролируемые поля;
- у задачи есть расширенная модель (не только текст).

Сейчас база для этого готова: голос может заполнять поле `text`, а дедлайн пользователь выбирает отдельно.

## 4) Поток данных на текущем этапе

1. Пользователь вводит текст и (опционально) дедлайн.
2. `AddTodo` вызывает `onAdd(text, deadline)`.
3. `App` формирует объект todo и добавляет его в `todos`.
4. `TodoItem` отрисовывает содержимое и даты.
5. При клике на чекбокс `TodoItem` вызывает `onToggleComplete`.
6. `App` обновляет `completed` в нужном элементе массива.

## 5) Технические заметки и улучшения

- В `onAdd` лучше перейти на функциональный `setTodos((prev) => [...prev, newTodo])`, чтобы избежать stale-state при быстрых добавлениях.
- Имена переменных стоит поправить для читаемости:
  - `todoToUpdata` -> `todoToUpdate`
  - `updataTodos` -> `updatedTodos`
- Поле `order` сейчас создаётся, но не используется для сортировки/drag-and-drop.
- `deadline` приходит строкой из `datetime-local`; если позже будет API, лучше нормализовать формат в UTC/ISO единообразно.

## 6) Что запомнить из части 2

- Модель данных todo должна расти вместе с требованиями UI.
- Состояние выполнения задачи правильнее хранить в родителе, а не в `TodoItem`.
- Форма с дополнительными полями (deadline) должна оставаться простой и контролируемой.
- Визуальные индикаторы времени (создано/просрочено) резко улучшают UX TODO-приложения.
