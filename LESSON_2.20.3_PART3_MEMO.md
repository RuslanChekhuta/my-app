# Памятка: Урок 2.20.3 (часть 3/10)
## Самое крутое TODO-приложение (с голосовым вводом)

## 1) Суть этапа 3

Главная цель этой части: сделать TODO-приложение «живым» и устойчивым к перезагрузке.

На этом этапе у вас уже реализовано:
- хранение задач в `localStorage`;
- синхронизация с удалённым API (`mockapi`);
- оптимистичные обновления UI (сразу показываем результат);
- откат состояния при ошибках сети.

Это важный фундамент перед голосовым вводом: голос добавит новый способ ввода, но данные уже должны надёжно сохраняться.

## 2) Что добавлено в коде

### В `src/App.jsx`
- константы:
  - `LOCAL_STORAGE_KEY = "todos"`
  - `API_URL = "https://69a86e4537caab4b8c61bab8.mockapi.io/api/v1/todos"`
- `useEffect` при старте:
  1. сначала читаются задачи из `localStorage`;
  2. затем пробуется загрузка с сервера;
  3. если сервер ответил успешно, список обновляется и снова сохраняется в `localStorage`.
- `onAdd`:
  - создаёт временную задачу с `temp_` id;
  - сразу добавляет её в UI;
  - отправляет `POST`;
  - после ответа сервера заменяет временную задачу на серверную.
- `toggleComplete`:
  - локально переключает `completed`;
  - отправляет `PUT` на сервер;
  - при ошибке откатывает состояние.
- `onDelete`:
  - локально удаляет задачу;
  - отправляет `DELETE`;
  - при ошибке возвращает предыдущий список.

## 3) Почему это правильно с точки зрения UX

- Пользователь не ждёт сеть, UI реагирует сразу.
- Если интернет медленный, ощущение приложения остаётся быстрым.
- При сбое данные не «ломаются» навсегда, потому что есть откат.

Именно так обычно строят современные фронтенд-приложения: optimistic UI + fallback.

## 4) Поток данных на этапе 3

1. Приложение стартует.
2. Показ данных из `localStorage` (быстрый рендер).
3. Фоновая загрузка с API и синхронизация.
4. Любое действие пользователя (`add/toggle/delete`) сначала меняет локальный state.
5. Затем отправляется запрос на сервер.
6. Если запрос успешен, фиксируем изменения в `localStorage`.
7. Если запрос упал, откатываем `todos`.

## 5) Что уже хорошо в вашем решении

- Есть разделение на «локальный state» и «удалённый источник».
- Добавление задачи учитывает ситуацию, когда сервер выдаёт свой `id`.
- В коде есть обработка ошибок (`try/catch`) для всех CRUD-операций.
- `TodoItem` использует централизованное состояние `completed`, а не локальный `useState`.

## 6) Технические риски, которые стоит держать в голове

- В `onAdd` используется `setTodos([...todos, newTodo])` и `order: todos.length + 1`.
  При очень быстрых последовательных действиях возможен stale state.

- В `catch` для `onAdd` откат делается через `setTodos(todos)`.
  Если параллельно были другие изменения, откат может вернуть не самое актуальное состояние.

- В `onAdd/toggleComplete/onDelete` не проверяется `response.ok` (кроме начальной загрузки).
  Лучше проверять статус ответа и бросать ошибку вручную при не-2xx.

- `JSON.parse(localStorage.getItem(...))` может упасть, если в хранилище битые данные.
  Желательно добавить безопасный `try/catch` вокруг парсинга.

## 7) Связь с голосовым вводом (следующие части)

Теперь архитектура готова к voice input:
- голос распознаёт фразу и заполняет `text` в `AddTodo`;
- `onAdd` уже умеет сохранять задачу локально + на сервер;
- значит голосовой ввод автоматически будет работать с той же надёжной цепочкой сохранения.

То есть в следующих частях останется добавить только слой распознавания речи, а не переписывать хранение данных.
